THE BASELINE

Singapore already tried.

Three official apps cover the normal case. None of them survives the abnormal one.

01 · SCDF + HTX
myResponder
Does: alerts 250K+ registered CFRs to cardiac arrest and minor fires within 800m–1.5km. In-app chat, photo/video to SCDF Ops, one-tap 995 with geolocation.
  • 100% dependent on mobile data — no offline fallback
  • Only cardiac + minor fires — no disaster coordination
  • Chat dies the moment towers degrade
  • No mesh, no store-and-forward
02 · MHA + HTX
SGSecure
Does: public reports suspicious activity (photo/video), receives emergency broadcasts, SMS-70999 Personal SOS, locates Civil Defence shelters.
  • Reporting needs internet — fails under network congestion
  • Passive one-way reporting, not two-way coordination
  • No confirmed offline mode, no MQTT, no mesh
  • Built for terrorism, not multi-hazard disaster ops
03 · SCDF · APR 2026
SG Alert
Does: cell-broadcast (CBS) alerts ride the radio network not mobile data, bypass DND, islandwide or geo-targeted, no app required.
  • Singtel-only during phased rollout · others get SMS
  • Older phones show wrong labels ("Presidential alert")
  • One-way broadcast — zero public feedback
  • Fails the moment cell towers are physically down

PILLAR 01 · WHEN THE NETWORK BREAKS

MQTT, not HTTP.

The single protocol decision that separates Kampung Kaki from every existing solution.

WHY MQTT
MQTT is designed for high-latency, low-bandwidth, intermittent links. HTTP is designed for a healthy data centre on a good day. In a disaster, the network is the patient.
  • QoS-1 + retained messages — the broker re-sends your SOS until the responder ACKs it
  • Store-and-forward — message queued locally, delivered the moment a link returns
  • Topic-tree dispatch — cases/{zone}/{caseId} scales to thousands of subscribers
  • Tiny wire format — 2-byte header, fits over LoRa-class backhaul if cell dies

PILLAR 03 · ONE INCIDENT, ONE OBJECT, ZERO HANDOFF

Three roles, one source of truth.

Today's stack splits a single emergency across three apps and three databases. Every handoff is manual, every reconciliation is lossy. We collapse that into one shared object — citizen, responder, and ops all reading and writing the same record.

CITIZEN
Raises the signal
Screenshot · feature-shots/00-shell-citizen-map.png — citizen map with SOS button, AI advisor, nearby alerts
Files REPORT-281 with category, GPS, evidence, severity. Same record, instantly visible to verifying ops and responders within range.
RESPONDER
Acts on the signal
Screenshot · feature-shots/11-responder-joinable-missions.png — Mission Board, fit-scored joinable cases
Reads REPORT-281 → joins CASE-281. Status, position, beats, AAR — all written back to the same record ops already has open.
OPS
Verifies, dispatches, broadcasts
Screenshot · feature-shots/19-ops-report-queue.png — Report queue, dispatch dock, zone tools
Sees corroboration count, declares EVENT-281, dispatches, draws a zone, broadcasts back to citizens — every action chained to the same incident ID.
WHAT THIS REPLACES
TODAY · 3 APPS, 3 DATABASES
SGSecure → phone call → SCDF Ops → SMS → myResponder
Citizen reports in app A. Ops re-keys it into system B. Dispatcher pages CFRs in app C. Responder writes outcome into a fourth radio log. Same incident, four data entries, four reconciliations, four chances to lose context.
KAMPUNG KAKI · 1 OBJECT
REPORT-281 → CASE-281 → EVENT-281 → AAR-281
One canonical incident object lives on the MQTT broker. Each role reads and writes its own slice. Every state change is timestamped, attributed, and immediately visible to the other two roles. Audit trail comes free.

UNDER THE HOOD

Boring stack, picked on purpose.

We chose mature, debuggable parts. Anything exotic gets caged behind a tier so it can't take the whole system down.

FRONTEND
  • React 18 · TypeScript 5 · Vite
  • TailwindCSS 3 · brutalist design tokens
  • MapLibre GL · OneMap raster tiles
  • Lucide icons
  • Role-aware single-page shell
  • Vercel Edge deploy · < 80 KB initial JS
BACKEND
  • FastAPI · Python 3.12
  • Redis · state + pub/sub channels
  • Mosquitto · MQTT 5 broker · QoS 1 retained
  • WebSocket bridge for browsers (MQTT-over-WS)
  • Vercel serverless · /api/live/* + /api/host/ask
AI · DATA · OBS
  • OpenAI GPT-5.5 · temperature 0.2
  • Grounding context capped at 4.8 KB
  • OneMap · LTA DataMall · NEA · data.gov.sg
  • RSS + Telegram (Tier A · no auth)
  • Source-health probe surface · per-tile age in seconds
DESIGN PRINCIPLES
1 · Never invent. Missing data renders as "unavailable" — never fabricated.
2 · Live or empty. If a feed is down, the panel says so; we never seed-fake.
3 · Role isolation. Citizen never sees ops-only widgets. Even in dev.
4 · One-handed UX. Every primary action is thumb-reachable on a phone.
5 · Brutalist clarity. Hard borders, big chips, zero ambiguity in an emergency.
6 · Source-traceable. Every AI reply chips out the tools it called.
